# terraform-provider-ipmitool v0.2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v0.2.0 of `markddavidoff/ipmitool` to the Terraform Registry,
folding in the consensus findings from the three-persona review: close the
fix-now security gaps (argv password leak, supply-chain hardening),
implement a coherent set of breaking schema changes (cipher required,
lockout-bypass env-var-only, WriteOnly user password), add Day-2 ergonomics
(ImportState, audit logging, health probe, concurrency cap), and harden the
release pipeline (split build/sign, SLSA provenance, SBOM).

**Architecture:** Single release cut. One v0.1 → v0.2 migration burden, not
two. The schema-breaking changes (cipher, force_lockout_risk removal,
user_password → user_password_wo, last_updated removal) land together.
Non-schema code changes (env-var password, tflog tracing, concurrency cap,
ImportState, partial-state recovery) layer on top. CI/release-pipeline
hardening lands in parallel since it doesn't touch the provider code.

**Tech Stack:** Go 1.25, `hashicorp/terraform-plugin-framework` (latest
v1.x), `goreleaser`, GitHub Actions, sops + age. New: Sigstore via
`actions/attest-build-provenance`, Syft via GoReleaser's `sboms` block,
govulncheck, gitleaks.

## Open questions

These are unresolved as of plan-writing. They block specific units; flagged
inline.

1. **Does `terraform-plugin-framework` WriteOnly graceful-degrade on TF
   <1.11?** Specifically: if an HCL file uses `user_password_wo` and the
   user is on TF 1.10, does the framework emit a clear error, or does it
   silently treat the attribute as a regular optional and write the value
   to state (worst-case)? Answered by POC 1.

2. **Does `ipmitool -E IPMI_PASSWORD` behave identically across the three
   packaged versions in our README install matrix (brew, apt, apk)?**
   ipmitool 1.8.18 is the README floor; the `-E` flag has been there since
   1.8.10 but env-var name variants exist in some forks. Answered by POC 2.

3. **Does the Registry accept GoReleaser output from a split-job workflow
   identically to the current single-job output?** Specifically: artifact
   filenames, signature path, manifest.json content. Answered by POC 3.

4. **What's the right ID format for `ImportState` on a per-resource basis?**
   `host:port/ch<N>/<entity>` is the working assumption but some resources
   (`ipmi_power`, `ipmi_chassis_identify`) have no channel/slot scope.
   Resolved in Unit 12.

5. **For the cipher-detect script's lockout warning: does the BMC reset its
   lockout counter on a successful authentication, or only after the
   lockout window expires?** Affects the script's recommendation to use a
   known-good password during probing. Answered during Unit 6.

**Why:** unblocks Unit 8 (WriteOnly user_password migration). High risk —
if WriteOnly fails on TF <1.11 in a way that writes the value to state, the

**Time-box:** 2h

- On TF 1.10 and earlier: framework either errors at plan time with a
  clear message OR persists value to state (which is the unsafe path).
- A documentation path exists for either behavior.

**Verdict template:**

### POC 3: Test-release end-to-end against the split build/sign workflow

**Why:** unblocks Unit 4 (release pipeline hardening). High risk — the
release flow currently works; breaking it during v0.2 means a botched
Registry publish. The Registry verifies GPG signature, SHA256 manifest,
and manifest.json structure. Any of the planned changes (job split,
artifact handoff via `upload-artifact`, SLSA attestation, SBOM) could
break one of those.

**Time-box:** 3h (longer because it requires a real CI run)
**Disposable:** partial — the workflow yaml IS the deliverable, but the
tag (`v0.1.1-test-rc1` etc.) and any test releases get cleaned up.
**Success criterion:**
1. `goreleaser release --rm-dist` succeeds in the split-job flow.
2. GPG signature on SHA256SUMS is valid (`gpg --verify` succeeds).
3. SLSA attestation is generated and verifiable (`gh attestation verify`).
4. SBOM `.spdx.json` is uploaded alongside zips.
5. Registry-style verification passes: download the zip, verify against
   the signed SHA256SUMS, parse manifest.json.

**Verdict template:**

| POC 3 outcome | Effect on plan |
|---|---|
| PASS — all five checks succeed | Unit 4 proceeds as planned. |
| PARTIAL — split-job + SBOM work, SLSA attestation breaks Registry | Unit 4 ships split-job + SBOM only; SLSA deferred to v0.3. |
| FAIL — any of split-job, GPG sign, or manifest verification breaks | Roll back to current single-job structure; ship v0.2 without H-8/H-9. Hold those for v0.3 with dedicated time. |

**Final outcome:** ✅ **PASS** (2026-06-17). Run #27711587987 with tag

- [ ] **Step 4.2: Rewrite `.github/workflows/release.yml`** as two jobs.
  Full file content:

```yaml
name: Release
on:
  push:
    tags:
      - 'v*'

permissions:
  contents: write
  id-token: write
  attestations: write

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
    outputs:
      tag: ${{ steps.tag.outputs.tag }}
    steps:
      - uses: actions/checkout@<sha>  # v6.x.y
        with:
          fetch-depth: 0
      - uses: actions/setup-go@<sha>  # v6.x.y
        with:
          go-version-file: 'go.mod'
      - id: tag
        run: echo "tag=${GITHUB_REF_NAME}" >> "$GITHUB_OUTPUT"
      - name: Build (snapshot, unsigned)
        uses: goreleaser/goreleaser-action@<sha>  # v7.x.y
        with:
          args: build --snapshot --clean
      - uses: actions/upload-artifact@<sha>  # v4.x.y
        with:
          name: dist
          path: dist/
          retention-days: 1

  sign:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      contents: write
      id-token: write
      attestations: write
    steps:
      - uses: actions/checkout@<sha>  # v6.x.y
        with:
          fetch-depth: 0
      - uses: actions/setup-go@<sha>  # v6.x.y
        with:
          go-version-file: 'go.mod'
      - uses: actions/download-artifact@<sha>  # v4.x.y
        with:
          name: dist
          path: dist/
      - name: Import GPG key
        id: import_gpg
        uses: crazy-max/ghaction-import-gpg@<sha>  # v7.x.y
        with:
          gpg_private_key: ${{ secrets.GPG_PRIVATE_KEY }}
          passphrase: ${{ secrets.PASSPHRASE }}
      - name: Release (sign + publish)
        uses: goreleaser/goreleaser-action@<sha>  # v7.x.y
        env:
          GPG_FINGERPRINT: ${{ steps.import_gpg.outputs.fingerprint }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          args: release --clean --skip=build
      - name: Generate SLSA provenance attestation
        uses: actions/attest-build-provenance@<sha>  # v3.x.y
        with:
          subject-path: 'dist/*.zip,dist/*SHA256SUMS*'
```

Note: `--skip=build` tells GoReleaser not to rebuild; it uses the
artifacts already in `dist/`. Verify this flag exists in the GoReleaser
version pinned — alternative is `--skip-build`.

set -euo pipefail

# detect-cipher.sh — probe an IPMI BMC for the highest-supported cipher.
#

# against the BMC. Most BMCs lock the account after 3–5 consecutive

    exit 2

host="$1"
user="$2"
pass="$3"
if [[ "$pass" == "-" ]]; then

    echo
fi

echo "==> Probing $host as $user (lockout-warning: see above)"
for c in 17 12 8 3 0; do
    printf "  cipher %2d: " "$c"
    if IPMI_PASSWORD="$pass" ipmitool -I lanplus -C "$c" -H "$host" -U "$user" -E mc info >/dev/null 2>&1; then

        echo
        echo "Add to your provider block:"
        echo "  cipher_suite = $c"
        exit 0
    else
        echo "FAIL"
    fi
done

echo
echo "ERROR: no supported cipher suite responded. Check host reachability, " >&2
echo "credentials, and that the BMC supports RMCP+ at all." >&2
exit 1

// AllowLockoutBypassed reports whether TF_IPMI_ALLOW_LOCKOUT=1 is set.
